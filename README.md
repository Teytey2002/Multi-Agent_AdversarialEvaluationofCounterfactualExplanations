# Multi-Agent Adversarial Evaluation of Counterfactual Explanations

This project studies how to evaluate the quality of **counterfactual explanations** for tabular machine learning models.  
The current stage of the project focuses on building the **predictive pipeline** and generating the first **counterfactual candidates** that will later be evaluated with metrics, explainability tools, and a multi-agent debate framework.

The long-term objective is to compare three evaluation strategies:

1. **Metrics-only** evaluation
2. **Single-LLM** evaluation
3. **Multi-agent adversarial debate**

At this stage, the repository already includes:

- dataset loading
- preprocessing for mixed tabular data
- training of multiple classifiers
- model comparison and persistence
- prediction on unfavorable cases
- first counterfactual generation with DiCE

## 1. Project Goal

The project investigates whether the quality of a counterfactual explanation can be assessed more robustly with a **structured adversarial reasoning process** than with classical metrics alone.

A counterfactual explanation answers a question of the form:

> *“What should change in this instance so that the model prediction flips?”*

For example, if a classifier predicts that an individual belongs to an unfavorable class, a counterfactual explanation proposes a nearby alternative profile that would change the prediction to the favorable class.

The final project is designed around four roles:

| Agent | Role | Objective |
|---|---|---|
| **Prosecutor** | Attacks the counterfactual | Detect fairness issues, unrealistic suggestions, or reliance on problematic features |
| **Defense** | Defends the counterfactual | Argue that the counterfactual is valid, actionable, and legitimate |
| **Expert Witness** | Provides technical evidence | Report quantitative metrics and explainability-based evidence |
| **Judge** | Synthesizes the debate | Produce a final structured verdict |

Before building this debate layer, a solid ML pipeline is required. This repository currently implements that first part.

## 2. Current Pipeline

The implemented pipeline currently follows the steps below:

1. **Load the Adult Income dataset** from OpenML.
2. **Preprocess the data** using a mixed numerical/categorical pipeline.
3. **Train three classification models**:
   - Logistic Regression
   - Random Forest
   - XGBoost
4. **Evaluate the models** on a held-out test set.
5. **Save the trained pipelines** as serialized `.joblib` files.
6. **Select a main predictive model** for the next stages.
7. **Run inference** on the dataset and identify unfavorable individuals.
8. **Sample a few unfavorable cases** and export them.
9. **Generate counterfactual explanations** for those cases using DiCE.
10. **Prepare the project for the next stages**: counterfactual metrics, SHAP analysis, and multi-agent evaluation.

In the current version, **XGBoost** is used as the main predictive model because it achieved the best overall performance among the three trained classifiers.

## 3. Dataset

### Adult Income dataset

The current implementation uses the **Adult Income** dataset, loaded directly from OpenML.

- **Source**: OpenML
- **Task**: binary classification
- **Objective**: predict whether a person earns more than 50K per year

### Target encoding

The original labels are converted into binary values:

- `<=50K` → `0`
- `>50K` → `1`

### Dataset characteristics

- **Number of samples**: `48,842`
- **Number of features**: `14`
- **Class distribution**:
  - class `0`: about `76.1%`
  - class `1`: about `23.9%`

This class imbalance is important because it means that **accuracy alone is not sufficient** to judge model quality. Precision, recall, and F1-score must also be considered.

## 4. Implemented Models

Three models are trained and compared.

### 4.1 Logistic Regression

Used as a simple and interpretable baseline.

**Advantages**:
- fast to train
- easy to interpret
- strong baseline for tabular classification

**Limitations**:
- linear decision boundary
- may miss more complex interactions in the data

---

### 4.2 Random Forest

An ensemble of decision trees trained with bagging.

**Advantages**:
- captures non-linear relationships
- robust on tabular datasets
- often strong out-of-the-box performance

**Limitations**:
- less interpretable globally
- can be weaker than boosting approaches on some tabular tasks

---

### 4.3 XGBoost

Gradient boosted decision trees.

**Advantages**:
- very strong performance on tabular data
- handles non-linear relationships well
- often a strong practical choice for structured data

**Limitations**:
- more complex than logistic regression
- explanations may need dedicated tools such as SHAP

## 6. Scripts Overview

### `src/data_loader.py`

Responsible for loading the dataset.

**Main responsibility**:
- fetch the Adult dataset from OpenML
- split features and target
- convert the target to binary labels

This script centralizes data access so that the dataset can easily be replaced later if needed.

---

### `src/preprocessing.py`

Builds the preprocessing pipeline.

Because the Adult dataset contains both numerical and categorical features, a `ColumnTransformer` is used.

#### Numerical preprocessing
- missing value imputation with the median
- feature scaling with `StandardScaler`

#### Categorical preprocessing
- missing value imputation with the most frequent value
- one-hot encoding with `OneHotEncoder(handle_unknown="ignore")`

This design ensures that preprocessing is learned on the training set and consistently reused at inference time.

---

### `src/models.py`

Defines and returns the three candidate classifiers:

- `logistic_regression`
- `random_forest`
- `xgboost`

This separation makes it easy to add, remove, or tune models without changing the rest of the pipeline.

---

### `src/utils.py`

Contains utility functions used across the project.

Typical responsibilities include:
- creating output directories
- evaluating model predictions
- saving serialized models
- saving metrics to JSON
- exporting summary tables to CSV

This avoids duplicating generic logic in the training script.

---

### `src/train.py`

This is the main training script.

#### What it does

1. creates output folders if needed
2. loads the Adult dataset
3. splits the data into train and test sets
4. builds the preprocessing pipeline
5. trains the three models
6. evaluates each model
7. saves each full trained pipeline as a `.joblib` file
8. saves detailed metrics in JSON format
9. saves a global comparison table in CSV format

#### Why full pipelines are saved

Each saved model contains both:
- the preprocessing pipeline
- the trained classifier

This is very convenient for later stages because the same object can be reused directly for:
- prediction
- probability estimation
- SHAP integration
- counterfactual generation

---

### `src/predict.py`

Used after training to run inference with the selected main model.

#### What it does

1. loads the saved main model (`xgboost.joblib`)
2. reloads the dataset
3. computes predictions and class probabilities
4. extracts individuals predicted as unfavorable (`prediction = 0`)
5. samples a small subset of those cases
6. saves them to `results/unfavorable_samples.csv`

The goal of this script is to prepare input instances for counterfactual generation.

---

### `src/generate_cf.py`

Uses DiCE to generate counterfactual explanations with basic realism constraints..

#### What it does

1. loads the trained predictive model
2. reloads the dataset to define the DiCE data object
3. loads the sampled unfavorable cases
4. defines the outcome variable for DiCE
5. specifies a restricted set of actionable features (e.g. workclass, occupation, hours-per-week, capital-related variables)
6. applies box constraints (permitted_range) to avoid unrealistic values (e.g. bounded capital-gain and working hours)
7. balances proximity vs diversity in the generation process through weighted parameters
8. applies post-hoc sparsity to limit the number of changes per counterfactual
9. generates multiple counterfactuals per instance
10. stores both the original instance and its counterfactuals in a structured format (with row_type and cf_rank)
11. saves them to `results/counterfactuals.csv`

This script is the first bridge between the predictive pipeline and the future evaluation framework.

---

### `src/cf_metrics.py`

Computes quantitative evaluation metrics for the generated counterfactual explanations, following the methodology described in the DiCE paper.

#### What it does

1. loads the generated counterfactual dataset (`counterfactuals.csv`)

2. separates **original instances** and their associated **counterfactuals** using `original_index` and `row_type`

3. identifies **continuous** and **categorical** features from the original dataset

4. computes **MAD (Median Absolute Deviation)** for continuous features to normalize distances

5. computes, for each instance:

   * **Validity**
     fraction of valid and unique counterfactuals

   * **Continuous proximity**
     normalized L1 distance (scaled by MAD) between original and counterfactuals

   * **Categorical proximity**
     proportion of unchanged categorical features

   * **Sparsity**
     proportion of features that remain unchanged

   * **Continuous diversity**
     pairwise distance between counterfactuals (continuous features)

   * **Categorical diversity**
     pairwise differences between counterfactuals (categorical features)

   * **Count-based diversity**
     fraction of features that differ between counterfactuals

6. aggregates the metrics:

   * per instance → `cf_metrics_per_instance.csv`
   * global averages → `cf_metrics_global.csv`


#### Notes

* correctly pairs each counterfactual with its **true original instance**
* follows the **evaluation protocol of the DiCE paper**
* distinguishes between **continuous and categorical distances**
* introduces **MAD-based normalization** for meaningful proximity computation
* adds proper **diversity metrics**, which are essential for evaluating sets of counterfactuals


#### Interpretation

The computed metrics allow to analyze trade-offs between:

* **validity vs realism**
* **proximity vs diversity**
* **sparsity vs actionability**

These outputs form the quantitative foundation for the next stage of the project:
the **Expert Witness agent**, which will rely on these metrics to provide structured, data-driven arguments in the multi-agent evaluation system.

## 11. Counterfactual Generation Logic

Counterfactuals are generated with **DiCE** from a subset of individuals predicted as unfavorable by the main model.

### Principle

For an instance predicted as class `0`, DiCE searches for alternative values of selected features such that the model prediction becomes class `1`.

### Current allowed features to vary

The first implementation allowed changes on features such as:
- `workclass`
- `education`
- `occupation`
- `hours-per-week`
- `capital-gain`
- `capital-loss`

This was intentionally permissive in order to obtain first working counterfactuals.

### Why this first version is useful

Even though the generated counterfactuals are not always realistic, they are already valuable because they reveal important model behavior.

For example, the generated explanations showed that:
- the model strongly exploits `capital-gain`
- some suggestions are unrealistic
- some outputs may contain missing values such as `nan`
- some changes are not necessarily actionable in the real world

This is not a failure of the project. On the contrary, it is exactly the kind of weakness that the later **evaluation framework** is supposed to detect and reason about.

## 12. Current Limitations

The current implementation is functional, but several limitations remain.

### 12.1 Counterfactual realism

Some generated counterfactuals are technically valid for the model but unrealistic for humans.

Examples observed in the first outputs:
- very large changes in `capital-gain`
- missing categorical values (`nan`)
- implausible feature combinations
- extreme changes in work hours

### 12.2 Actionability constraints are still basic

The first version does not yet fully enforce which features should or should not be changed.

For example, in a fairness-sensitive setting, features like:
- `race`
- `sex`
- `age`

should typically be protected or fixed.

### 12.3 Counterfactual metric computation is not finalized

The first metric script needs to be revised so that each generated counterfactual is compared against its correct original instance.

### 12.4 Explainability integration not yet implemented

SHAP is planned but not yet added in the current stage.
